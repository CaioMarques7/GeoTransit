# ADR-005 — Adotar uma API Orientada à Consulta (Thin Query API)

**Status:** Accepted

**Data:** *A definir*

**Relacionamentos:**

* ADR-001 — Adotar uma Arquitetura Orientada à Leitura
* ADR-002 — Separar o Processamento Offline do Processamento Online
* ADR-003 — Separar as Projeções Espaciais das Projeções Operacionais
* ADR-004 — Introduzir um Prediction Engine para Processamento do GTFS Realtime
* Visão Arquitetural
* Modelo de leitura

---

# Contexto

O principal objetivo da plataforma é responder consultas geoespaciais com baixa latência e alta previsibilidade de desempenho.

Na arquitetura original, a API acumulava diversas responsabilidades durante o atendimento de cada requisição:

* localizar as paradas mais próximas;
* identificar as linhas atendidas por essas paradas;
* consultar dados do GTFS Realtime;
* correlacionar informações estruturais e operacionais;
* calcular a ordem de chegada dos veículos;
* construir a resposta para o consumidor.

Embora funcional, essa abordagem fazia com que a API executasse processamento proporcional ao volume de requisições recebidas.

Além de aumentar a latência, essa arquitetura dificultava a evolução do sistema, pois qualquer alteração na lógica de negócio exigia modificações na própria API.

---

# Decisão

A API será tratada exclusivamente como uma camada de consulta (**Query API**).

Sua responsabilidade será:

* receber requisições dos consumidores;
* consultar os modelos de leitura apropriados;
* compor a resposta utilizando informações previamente materializadas;
* devolver o resultado ao cliente.

A API não será responsável por:

* processar arquivos GTFS;
* interpretar eventos do GTFS Realtime;
* calcular estimativas de chegada;
* construir projeções;
* executar processamento geoespacial complexo;
* manter estado operacional do sistema.

Toda lógica de processamento permanecerá concentrada nos pipelines especializados definidos pelos ADRs anteriores.

---

# Justificativa

O principal objetivo desta decisão é manter a API simples, previsível e altamente escalável.

Ao limitar suas responsabilidades ao atendimento das consultas, reduz-se significativamente o custo computacional por requisição e evita-se que regras de negócio sejam duplicadas ou distribuídas entre diferentes componentes.

Essa abordagem também permite que novos consumidores utilizem a plataforma sem conhecer detalhes sobre GTFS, algoritmos de previsão ou mecanismos internos de processamento.

A API passa a atuar como um contrato estável entre a plataforma e seus consumidores.

---

# Consequências

## Positivas

* Redução significativa da complexidade da API.
* Baixa latência e maior previsibilidade das consultas.
* Escalabilidade horizontal simplificada.
* Facilidade para evoluir pipelines internos sem alterar contratos públicos.
* Separação clara entre processamento e disponibilização de informações.
* Menor acoplamento entre consumidores e regras de negócio.

## Negativas

* Dependência da disponibilidade dos modelos de leitura.
* Necessidade de manter projeções continuamente atualizadas.
* A API deixa de possuir capacidade para reconstruir respostas caso as projeções estejam indisponíveis.
* Maior importância do monitoramento dos pipelines responsáveis pela atualização dos dados.

---

# Alternativas Consideradas

## Centralizar toda a lógica de negócio na API

Consistia em utilizar a API como responsável tanto pelo processamento quanto pela disponibilização das informações.

### Motivo da rejeição

Essa abordagem aumenta o custo computacional de cada requisição, reduz a capacidade de escalabilidade da plataforma e dificulta a evolução das regras de negócio.

---

## Permitir que a API execute cálculos complementares quando necessário

Consistia em manter a maior parte do processamento nos pipelines, permitindo que a API realizasse cálculos adicionais em situações específicas.

### Motivo da rejeição

Essa alternativa cria exceções ao modelo arquitetural e tende a concentrar novamente regras de negócio na camada de consulta ao longo da evolução do sistema.

Além disso, dificulta a previsibilidade do desempenho das requisições.

---

## Expor diretamente os modelos de leitura aos consumidores

Consistia em permitir que aplicações clientes consultassem diretamente os mecanismos de armazenamento das projeções.

### Motivo da rejeição

Essa abordagem aumenta o acoplamento entre consumidores e infraestrutura, dificulta a evolução da arquitetura e elimina a possibilidade de manter contratos estáveis para integração.

---

# Impacto na Arquitetura

Esta decisão estabelece a API como o único ponto de entrada para consultas da plataforma.

A arquitetura passa a ser organizada em quatro responsabilidades bem definidas:

* **Pipeline Offline** — produz conhecimento estrutural.
* **Prediction Engine** — produz conhecimento operacional.
* **Read Models** — armazenam conhecimento especializado para consulta.
* **Query API** — disponibiliza esse conhecimento aos consumidores.

Essa separação reduz o acoplamento entre componentes e reforça o princípio de processamento antecipado adotado pela arquitetura.

---

# Relação com outros ADRs

Esta decisão depende diretamente dos princípios estabelecidos nos ADRs anteriores.

Ela consolida a arquitetura orientada à leitura ao impedir que responsabilidades de processamento retornem para a camada de atendimento das requisições.

Os ADRs subsequentes deverão preservar essa separação de responsabilidades ao definir tecnologias de armazenamento, estratégias de atualização e mecanismos de persistência.
