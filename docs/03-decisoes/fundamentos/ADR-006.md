# ADR-006 — Atualizar os Modelos de Leitura do GTFS Static por Publicação Atômica

**Status:** Accepted

**Data:** *A definir*

**Relacionamentos:**

* ADR-001 — Adotar uma Arquitetura Orientada à Leitura
* ADR-002 — Separar o Processamento Offline do Processamento Online
* ADR-003 — Separar as Projeções Espaciais das Projeções Operacionais
* Read Model Design

---

# Contexto

Os dados provenientes do GTFS Static representam a estrutura do sistema de transporte público.

Embora sua frequência de atualização seja baixa, a construção das projeções espaciais pode demandar tempo significativo, especialmente em municípios com grande volume de paradas, linhas e viagens.

Na solução original, o processo de atualização reconstruía diretamente os dados utilizados pela API.

Durante esse período, parte das informações ainda não estava disponível, ocasionando degradação temporária do serviço e obrigando a API a recorrer ao banco relacional como mecanismo de fallback.

Essa estratégia acoplava a disponibilidade da API ao tempo necessário para reconstrução das projeções.

---

# Decisão

A atualização das projeções derivadas do GTFS Static será realizada em duas etapas distintas:

1. construção completa de uma nova versão dos modelos de leitura;
2. publicação atômica da nova versão para consumo pela API.

Enquanto a nova versão estiver sendo construída, a versão atualmente publicada continuará atendendo normalmente às consultas.

Somente após a conclusão do processamento e das validações necessárias a nova versão será promovida para produção.

A API sempre consultará apenas versões completas e consistentes dos modelos de leitura.

---

# Justificativa

A construção das projeções é uma atividade de processamento offline e não deve impactar a disponibilidade do serviço de consulta.

Ao separar construção e publicação, evita-se que consumidores acessem modelos parcialmente carregados ou inconsistentes.

Essa abordagem também reduz a necessidade de mecanismos de fallback durante atualizações planejadas e permite validar a integridade das projeções antes de disponibilizá-las para produção.

---

# Consequências

## Positivas

* Eliminação de indisponibilidade causada pelo recarregamento dos modelos de leitura.
* Publicação apenas de versões completas e consistentes.
* Redução da dependência de mecanismos de fallback durante atualizações.
* Possibilidade de validar os modelos antes da publicação.
* Facilidade para realizar rollback para uma versão anterior em caso de falha.
* Melhor previsibilidade operacional durante a atualização do GTFS Static.

## Negativas

* Necessidade de manter simultaneamente duas versões dos modelos durante o processo de atualização.
* Maior consumo temporário de recursos de armazenamento.
* Introdução de um mecanismo adicional para controle de versões publicadas.

---

# Alternativas Consideradas

## Atualizar os modelos diretamente em produção

Consistia em substituir gradualmente os dados utilizados pela API à medida que o processamento fosse concluído.

### Motivo da rejeição

Essa abordagem expõe modelos parcialmente atualizados aos consumidores e torna a disponibilidade da API dependente da duração do processamento.

---

## Interromper o serviço durante a atualização

Consistia em suspender temporariamente as consultas enquanto as projeções fossem reconstruídas.

### Motivo da rejeição

Essa alternativa reduz a disponibilidade da plataforma e não atende aos requisitos de operação contínua.

---

## Utilizar apenas um mecanismo de fallback para o banco relacional

Consistia em manter o comportamento da solução original, recorrendo ao SQL Server sempre que as projeções ainda não estivessem completamente carregadas.

### Motivo da rejeição

Embora preserve a disponibilidade, essa abordagem mantém dependência do banco relacional justamente nos momentos de maior custo operacional e não elimina a causa do problema.

---

# Impacto na Arquitetura

Esta decisão estabelece que os modelos de leitura são publicados como unidades consistentes.

O Pipeline Offline deixa de modificar diretamente os dados utilizados pela API e passa a produzir versões completas que somente serão disponibilizadas após sua conclusão.

Essa estratégia desacopla o tempo de processamento do tempo de disponibilidade da plataforma, preservando a estabilidade das consultas durante atualizações do GTFS Static.

---

# Decisões de Implementação

Este ADR não determina como a publicação atômica será implementada.

A solução poderá utilizar diferentes estratégias, desde que preserve o seguinte princípio:

> **A API nunca deve consultar modelos de leitura parcialmente atualizados.**

Exemplos de implementações compatíveis incluem:

* troca de versão por ponteiro lógico;
* namespaces versionados;
* estratégias Blue/Green;
* troca atômica de chaves;
* mecanismos equivalentes de publicação.

---

# Relação com outros ADRs

Esta decisão complementa o Pipeline Offline definido no ADR-002 e garante que as projeções espaciais definidas no ADR-003 sejam disponibilizadas de forma consistente para a Query API.

Ela também reduz a necessidade de dependência operacional do sistema de referência durante atualizações planejadas.
