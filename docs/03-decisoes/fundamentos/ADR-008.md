# ADR-008 — Manter um Sistema de Persistência como Fonte Canônica dos Dados

**Status:** Accepted

**Data:** *A definir*

**Relacionamentos:**

* ADR-001 — Adotar uma Arquitetura Orientada à Leitura
* ADR-003 — Separar as Projeções Espaciais das Projeções Operacionais
* ADR-006 — Atualizar os Modelos de Leitura do GTFS Static por Publicação Atômica
* ADR-007 — Utilizar um Armazenamento Especializado para Consultas Geoespaciais
* Modelo de leitura

---

# Contexto

A arquitetura proposta utiliza modelos de leitura especializados para atender consultas de alta frequência com baixa latência.

Esses modelos são derivados dos dados oficiais publicados pelos operadores de transporte e produzidos pelos pipelines da plataforma.

Embora sejam fundamentais para o desempenho da solução, os modelos de leitura não representam a fonte oficial dos dados.

Eles podem ser reconstruídos a qualquer momento a partir das fontes originais.

Por esse motivo, a arquitetura necessita de um mecanismo responsável por preservar os dados processados e servir como referência para reconstruções, auditorias, validações e recuperação operacional.

---

# Decisão

A plataforma manterá um sistema de persistência responsável por armazenar os dados canônicos utilizados pelos pipelines de processamento.

Esse sistema será considerado a **fonte de verdade** (*System of Record*) da aplicação.

Na implementação desta arquitetura, esse papel será desempenhado pelo **SQL Server**.

Os modelos de leitura serão tratados como representações derivadas, descartáveis e reconstruíveis.

A API não consultará diretamente o sistema de persistência durante sua operação normal.

---

# Justificativa

A separação entre persistência e consulta é um dos princípios fundamentais da arquitetura orientada à leitura.

Os modelos de leitura são otimizados para desempenho e podem ser reconstruídos sempre que necessário.

Já o sistema de persistência possui responsabilidades diferentes:

* preservar os dados processados;
* permitir reprocessamentos;
* suportar auditorias;
* servir de base para reconstrução das projeções;
* garantir consistência durante atualizações do GTFS Static.

Essa separação reduz o acoplamento entre armazenamento e consulta, permitindo que cada componente evolua de acordo com sua responsabilidade.

---

# Consequências

## Positivas

* Definição clara da fonte canônica dos dados.
* Facilidade para reconstrução completa dos modelos de leitura.
* Redução do risco de perda de dados derivados.
* Maior confiabilidade durante reprocessamentos.
* Suporte a auditoria e validação dos pipelines.
* Independência entre mecanismos de persistência e mecanismos de consulta.

## Negativas

* Introdução de um mecanismo adicional de armazenamento.
* Necessidade de sincronização entre persistência e projeções.
* Maior custo operacional em comparação com uma arquitetura baseada em um único banco de dados.
* Necessidade de monitoramento dos processos de atualização entre os componentes.

---

# Alternativas Consideradas

## Utilizar apenas os modelos de leitura como armazenamento permanente

Consistia em manter exclusivamente as projeções utilizadas pela API.

### Motivo da rejeição

Os modelos de leitura são especializados para consulta e não preservam necessariamente todas as informações necessárias para reconstrução, auditoria ou evolução da plataforma.

Além disso, qualquer corrupção das projeções exigiria novo processamento diretamente a partir das fontes externas.

---

## Consultar diretamente os arquivos GTFS sempre que necessário

Consistia em utilizar os arquivos publicados pelos operadores como única fonte de reconstrução.

### Motivo da rejeição

Essa abordagem aumenta a dependência de sistemas externos, dificulta auditorias e reduz a previsibilidade operacional da plataforma.

Também impede o armazenamento de informações derivadas produzidas durante os pipelines.

---

## Eliminar completamente o banco relacional

Consistia em concentrar persistência e consulta exclusivamente no mecanismo especializado utilizado pelas projeções.

### Motivo da rejeição

Essa alternativa mistura responsabilidades de armazenamento permanente e consulta de alta performance, reduzindo a flexibilidade arquitetural e dificultando futuras evoluções dos modelos de leitura.

---

# Impacto na Arquitetura

Esta decisão estabelece uma separação definitiva entre persistência e consulta.

O sistema de persistência passa a representar o repositório oficial dos dados processados pela plataforma.

Os modelos de leitura passam a ser considerados artefatos derivados, cuja existência é justificada exclusivamente pela necessidade de atender consultas com baixa latência.

Essa separação fortalece a resiliência da arquitetura, permitindo reconstruções completas das projeções sem perda de informações.

---

# Decisões de Implementação

Este ADR registra o SQL Server como implementação do sistema de persistência.

Entretanto, o princípio arquitetural permanece independente da tecnologia utilizada.

Qualquer mecanismo capaz de desempenhar o papel de **System of Record**, oferecendo persistência confiável e suporte aos pipelines de processamento, poderá substituir a implementação atual sem alterar os fundamentos da arquitetura.

---

# Relação com outros ADRs

Esta decisão complementa a arquitetura orientada à leitura estabelecida no ADR-001, preserva a separação entre processamento e consulta introduzida pelos ADRs subsequentes e define o papel do sistema de persistência dentro da plataforma.

Enquanto o ADR-007 especializa o mecanismo de consulta, este ADR especializa o mecanismo de persistência, concluindo a separação de responsabilidades entre os componentes responsáveis por armazenar e disponibilizar as informações.
